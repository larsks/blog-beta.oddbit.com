Odd Bitshttp://blog-beta.oddbit.com/2010-05-11T00:00:00-04:00Pushing a Git repository to Subversion2010-05-11T00:00:00-04:00Lars Kellogg-Stedmantag:blog-beta.oddbit.com,2010-05-11:2010/05/11/pushing-git-repository-to-subversion/<p>I recently set up a git repository server (using <a href="http://scie.nti.st/2007/11/14/hosting-git-repositories-the-easy-and-secure-way">gitosis</a> and <a href="https://git.wiki.kernel.org/index.php/Gitweb">gitweb</a>). Among the required features of the system was the ability to publish the git repository to a read-only Subversion repository. This sounds simple in principle but in practice proved to be a bit tricky.</p> <p>Git makes an excellent Subversion client. You can use the git svn ... series of commands to pull a remote Subversion repository into a local git working tree and then have all the local advantages of git forcing the central code repository to change version control software. An important aspect of this model is that:</p> <ul> <li>The Subversion repository is the primary source of the code, and</li> <li>You populate your local git repository by pulling from the remote Subversion repository.</li> </ul> <p>It is possible to push a git change history into an empty Subversion repository. Most instructions for importing a git repository look something like this, and involve replaying your git change history on top of the Subversion change history:</p> <ul> <li>svn mkdir $REPO/{trunk, tags, branches}</li> <li>git svn init -s $REPO</li> <li>git svn fetch</li> <li>git rebase trunk</li> <li>git svn dcommit</li> </ul> <p>This works, and is fine as long as there are no other clones of your git repository out there. The mechanism outlined here has a fatal flaw: it modifies the change history of the <em>master</em> branch. If you were working in a clone of a remote git repository and you were to run git status after the above steps, you would see something like:</p> <div class="highlight"><pre><span class="c"># On branch master</span> <span class="c"># Your branch and &#39;origin/master&#39; have diverged,</span> <span class="c"># and have 3 and 2 different commit(s) each, respectively.</span> </pre></div> <p>If you were then to try to push this to the remote repository, you would get an error:</p> <div class="highlight"><pre><span class="err">$</span> <span class="n">git</span> <span class="n">push</span> <span class="n">To</span> <span class="p">...</span><span class="o">/</span><span class="n">myrepo</span><span class="o">:</span> <span class="o">!</span> <span class="p">[</span><span class="n">rejected</span><span class="p">]</span> <span class="n">master</span> <span class="o">-&gt;</span> <span class="n">master</span> <span class="p">(</span><span class="n">non</span><span class="o">-</span><span class="n">fast</span> <span class="n">forward</span><span class="p">)</span> <span class="nl">error:</span> <span class="n">failed</span> <span class="n">to</span> <span class="n">push</span> <span class="n">some</span> <span class="n">refs</span> <span class="n">to</span> <span class="err">&#39;</span><span class="p">...</span><span class="o">/</span><span class="n">myrepo</span><span class="err">&#39;</span> </pre></div> <p>In cases where the git change history is shared with other git repositories, we need a solution that does not modify the <em>master</em> branch. We can get this my modifying the procedure slightly.</p> <p>The initial sequence is still the same:</p> <ul> <li>svn mkdir $REPO/{trunk, tags, branches}</li> <li>git svn init -s $REPO</li> <li>git svn fetch</li> </ul> <p>But instead of rebasing onto the <em>master</em> branch, we create a local branch for managing the synchronization:</p> <ul> <li>git checkout -b svnsync</li> <li>git rebase trunk</li> <li>git svn dcommit</li> </ul> <p>At this point we have changed the history of the <em>svnsync</em> branch and we have left the <em>master</em> branch untouched. Subsequent updates look like this:</p> <ul> <li>git checkout master</li> <li>git pull</li> <li>git checkout svnsync</li> <li>git rebase master</li> <li>git rebase trunk</li> <li>git svn dcommit</li> </ul> <p>This gives us what we want: we can publish our git repository to a Subversion repository while maintaining the shared change history among our existing git clones.</p>
