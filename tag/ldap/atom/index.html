Most of these guides solve the problem of authentication by embedding a username and password into a configuration file somewhere on your system. While this works, it presents some problems:</p> <ul> <li>If you use a common account for authentication from all of your Linux systems, a compromise on one system means updating the configuration of all of your systems.</li> <li>If you don't want to use a common account, you need to provision a new account for each computer...</li> <li>...which is silly, because if you join the system to Active Directory there is already a computer object associated with the system that can be used for authentication.</li> </ul> <p>This document describes how to configure a Linux system such that queries generated by <a href="http://www.padl.com/OSS/nss_ldap.html">nss_ldap</a> will use either the current user's Kerberos credentials, or, for the root user, credentials stored in a Kerberos credentials cache.</p> <ul> <li> <p>Your Linux system must have a valid <code>keytab</code> file.</p> <p>A <code>keytab</code> is a file containing pairs of Kerberos principals and encrypted keys.</p> <p>Joining Active Directory using Samba's <code>net ads join</code> will create the necessary keytab. It is also possible to create the keytab on your Windows domain controller and install it on your Linux systems. Instructions for doing this are beyond the scope of this document.</p> </li> <li> <p>Host objects in Active Directory must have a <code>userPrincipalName</code> attribute.</p> <p>For example:</p> <div class="highlight"><pre><span class="err">$</span> <span class="n">ldapsearch</span> <span class="n">cn</span><span class="o">=</span><span class="n">dottiness</span> <span class="n">userPrincipalName</span> <span class="nl">dn:</span> <span class="n">CN</span><span class="o">=</span><span class="n">DOTTINESS</span><span class="p">,</span><span class="n">CN</span><span class="o">=</span><span class="n">Computers</span><span class="p">,</span><span class="n">dc</span><span class="o">=</span><span class="n">example</span><span class="p">,</span><span class="n">dc</span><span class="o">=</span><span class="n">com</span> <span class="nl">userPrincipalName:</span> <span class="n">host</span><span class="o">/</span><span class="n">dottiness</span><span class="p">.</span><span class="n">example</span><span class="p">.</span><span class="n">com</span><span class="err">@</span><span class="n">EXAMPLE</span><span class="p">.</span><span class="n">COM</span> </pre></div> <p>Active Directory <em>does not</em> automatically create a <code>userPrincipalName</code> when a new host object is provisioned. You will either need to provide this value manually or develop an automated process that will populate this field when provisioning new host objects.</p> </li> </ul> <p>Kerberos credentials have a maximum usable lifetime. The cached credentials used for root queries by <code>nss_ldap</code> must be refreshed periodically in order to function.</p> <p>You will need to install a crontab (e.g., in <code>/etc/cron.d</code>) While this makes the proxy a single point of failure, it is (a) a very simple tool and thus less prone to complex failure modes, (b) running on the same host as the web application, and (c) is completely under our control.</p> <p>For this example, I will use <a href="http://www.inlab.de/balance.html">Balance</a>, a simple TCP load balancer from <a href="http://www.inlab.de/">Inlab Software GmbH</a>. There are packages available for most major Linux distributions, including <a href="http://fedoraproject.org/">Fedora</a> and <a href="http://www.centos.org/">CentOS</a>.</p> <p>Balance is configured completely on the command line. To provide round-robin access to a suite of three directory servers running LDAP over SSL, you might use the following command line:</p> <div class="highlight"><pre><span class="n">balance</span> <span class="o">-</span><span class="n">b</span> <span class="mf">127.0.0.1</span> <span class="mi">636</span> <span class="mf">10.1.1.1</span> <span class="mf">10.1.1.2</span> </pre></div> <p>Using balance's terminology, this creates one <em>group</em> of two <em>channels</em>. Balance will round-robin among the channels in this group. Note that here and in subsequent examples we are binding the proxy to the loopback interface so that it is only available to applications running on the same host.</p> <p>If you would prefer to preferentially send all the requests to the first server, and only use the second server if the first is unavailable, you could use a configuration like this:</p> <div class="highlight"><pre><span class="n">balance</span> <span class="o">-</span><span class="n">b</span> <span class="mf">127.0.0.1</span> <span class="mi">636</span> <span class="mf">10.1.1.1</span> <span class="err">\</span><span class="o">!</span> <span class="mf">10.1.1.2</span> </pre></div> <p>While you can run balance from a standard init (/etc/rc.d/...) script, I prefer to use a service manager such as <a href="http://smarden.org/runit/">runit</a> which takes care of restarting the service if it should exit unexpectedly. You could achieve the same thing in a slightly less flexible fashion by putting your balance command line in /etc/inittab. In either case you need to add the -f option to the command line, which causes balance to stay in the foreground.</p> <h1>Problem 2: Debugging LDAP over SSL</h1> <h2>The problem</h2> <p>It is convenient to use a packet tracer such as <a href="http://www.wireshark.org/">Wireshark</a> to debug LDAP protocol errors. Complete documentation for the meta backend can be found in the <a href="http://www.openldap.org/software/man.cgi?query=slapd-meta&amp;apropos=0&amp;sektion=0&amp;manpath=OpenLDAP+2.4-Release&amp;format=html">slapd-meta man page</a>.</p> <p>Put the following into a file called slapd-frontend.conf (we'll discuss the details in moment):</p> <div class="highlight"><pre><span class="n">database</span> <span class="n">meta</span> <span class="n">suffix</span> <span class="s">&quot;dc=example,dc=com&quot;</span> <span class="n">uri</span> <span class="s">&quot;ldap://localhost/ou=backend1,dc=example,dc=com&quot;</span> <span class="n">suffixmassage</span> <span class="s">&quot;ou=backend1,dc=example,dc=com&quot;</span> <span class="s">&quot;ou=backend1&quot;</span> <span class="n">uri</span> <span class="s">&quot;ldap://localhost/ou=backend2,dc=example,dc=com&quot;</span> <span class="n">suffixmassage</span> <span class="s">&quot;ou=backend2,dc=example,dc=com&quot;</span> <span class="s">&quot;ou=backend2&quot;</span> </pre></div> <p>And then add to slapd.conf:</p> <div class="highlight"><pre><span class="n">include</span> <span class="o">/</span><span class="n">etc</span><span class="o">/</span><span class="n">openldap</span><span class="o">/</span><span class="n">slapd</span><span class="o">-</span><span class="n">frontend</span><span class="p">.</span><span class="n">conf</span> </pre></div> <p>Restart slapd. Let's do a quick search to see exactly what we've accomplished:</p> <div class="highlight"><pre><span class="err">$</span> <span class="n">ldapsearch</span> <span class="o">-</span><span both backends, merged into the dc=example,dc=com hierarchy.</p> <h2>A closer look</h2> <p>Let's take a closer look at the meta backend configuration.</p> <div class="highlight"><pre><span class="n">database</span> <span class="n">meta</span> <span class="n">suffix</span> <span class="s">&quot;dc=example,dc=com&quot;</span> </pre></div> <p>The database statement begins a new database definition. The suffix statement identifies the namespace that will be served by this particular database.</p> <p>Here is the proxy for backend1 (the entry for backend2 is virtually identical):</p> <div class="highlight"><pre><span class="n">uri</span> <span class="s">&quot;ldap://localhost/ou=backend1,dc=example,dc=com&quot;</span> <span class="n">suffixmassage</span> <span class="s">&quot;ou=backend1,dc=example,dc=com&quot;</span> <span class="s">&quot;ou=backend1&quot;</span> </pre></div> <p>The uri statement defines the host (and port) serving the target directory tree. The full syntax of the uri statement is described in the <a href="http://www.openldap.org/software/man.cgi?query=slapd-meta&amp;apropos=0&amp;sektion=0&amp;manpath=OpenLDAP+2.4-Release&amp;format=html">slapd-meta man page</a>; what we have here is a very simple example. The <em>naming context</em> of the URI must fall within the namespace defined in the suffix statement at the beginning of the database definition.</p> <p>The suffixmassage statement performs simple rewriting on distinguished names. It directs <em>slapd</em> to replace ou=backend1,dc=example,dc=com with ou=backend1 when communicating with the backend directory (and vice-versa).</p> <p>You can perform simple rewriting of attribute and object classes with the map statement. For example, if backend1 used a sAMAccountName attribute and our application was expecting a uid attribute, we could add this after the suffixmassage statement:</p> <div class="highlight"><pre><span class="n">map</span> <span class="n">attribute</span> <span class="n">uid</span> <span class="n">sAMAccountName</span> </pre></div> <h1>Conclusion</h1> <p>The sample configuration files, data, and code referenced in this post are available online in <a href="http://github.com/larsks/OpenLDAP-Metadirectory-Example">a github repository</a>:</p> <blockquote> <p><a href="http://github.com/larsks/OpenLDAP-Metadirectory-Example">http://github.com/larsks/OpenLDAP-Metadirectory-Example</a></p> </blockquote> <p>I hope you've found this post useful, or at least informative.
