When using an agent -- particularly with agent forwarding enabled -- this allows you to authenticate to a remote host without having to (a) repeatedly type in your password or (b) expose an unencrypted private key to remote systems.</p> <p>If one is temtped to use SSH keys as authentication credentials outside of ssh, one would ideally be able to take advantage of the ssh agent for these same reasons.</p> <p>This article discusses what is required to programmatically interact with the agent and with the OpenSSL libraries for signing data and verifying signatures.</p> <h4>Signing data with ssh-agent</h4> <p>The SSH agent does not provide clients with direct access to an unencrypted private key. Rather, it will accept data from the client and return the signature of the SHA1 hash of the data.</p> <p>The agent communicates over a unix socket using the <a href="http://www.openbsd.org/cgi-bin/cvsweb/src/usr.bin/ssh/PROTOCOL.agent?rev=HEAD;content-type=text%2Fplain">ssh agent protocol</a> defined in <a href="http://www.openbsd.org/cgi-bin/cvsweb/src/usr.bin/ssh/authfd.h?rev=HEAD;content-type=text%2Fplain">authfd.h</a>. This can lead to unnecessary service outages if your directory service infrastructure is highly available (e.g., you are running Active Directory) and your application cannot take advantage of this fact.</p> <h2>A solution</h2> <p>We can provide a level of redundancy by passing the LDAP connections through a load balancing proxy. While this makes the proxy a single point of failure, it is (a) a very simple tool and thus less prone to complex failure modes, (b) running on the same host as the web application, and (c) is completely under our control.</p> <p>For this example, I will use <a href="http://www.inlab.de/balance.html">Balance</a>, a simple TCP load balancer from <a href="http://www.inlab.de/">Inlab Software GmbH</a>. There are packages available for most major Linux distributions, including <a href="http://fedoraproject.org/">Fedora</a> and <a href="http://www.centos.org/">CentOS</a>.</p> <p>Balance is configured completely on the command line. To provide round-robin access to a suite of three directory servers running LDAP over SSL, you might use the following command line:</p> <div class="highlight"><pre><span class="n">balance</span> <span class="o">-</span><span class="n">b</span> <span class="mf">127.0.0.1</span> <span class="mi">636</span> <span class="mf">10.1.1.1</span> <span class="mf">10.1.1.2</span> </pre></div> <p>Using balance's terminology, this creates one <em>group</em> of two <em>channels</em>. Balance will round-robin among the channels in this group. Note that here and in subsequent examples we are binding the proxy to the loopback interface so that it is only available to applications running on the same host.</p> <p>If you would prefer to preferentially send all the requests to the first server, and only use the second server if the first is unavailable, you could use a configuration like this:</p> <div class="highlight"><pre><span class="n">balance</span> <span class="o">-</span><span class="n">b</span> <span class="mf">127.0.0.1</span> <span class="mi">636</span> <span class="mf">10.1.1.1</span> <span class="err">\</span><span class="o">!</span> <span class="mf">10.1.1.2</span> </pre></div> <p>While you can run balance from a standard init (/etc/rc.d/...) script, I prefer to use a service manager such as <a href="http://smarden.org/runit/">runit</a> which takes care of restarting the service if it should exit unexpectedly. You could achieve the same thing in a slightly less flexible fashion by putting your balance command line in /etc/inittab. In either case you need to add the -f option to the command line, which causes balance to stay in the foreground.</p> <h1>Problem 2: Debugging LDAP over SSL</h1> <h2>The problem</h2> <p>It is convenient to use a packet tracer such as <a href="http://www.wireshark.org/">Wireshark</a> to debug LDAP protocol errors. This is often more informative than the debugging information that will be available to you on the client side, and may be more useful than server side debugging in many cases, even supposing that you have administrative access to the directory servers.</p> <h2>A solution</h2> <p>You can use <a href="http://www.stunnel.org/">Stunnel</a>, a general purpose SSL proxy tool, to intercept unencrypted client connections on the local machine and then forward them over an SSL channel to a remote server. This makes the unencrypted LDAP traffic available on the loopback interface while still ensuring that it is encrypted on the wire.</p> <p>Stunnel can operate both as an SSL server and as an SSL client. In this case, we will be running it in client mode, connecting to a remote SSL server (or to the proxy configured in our previous example). Stunnel is configured by means of a simple INI-style configuration file. To achieve the goals of this example we might put the following configuration in a file (say, stunnel.conf):</p> <div class="highlight"><pre><span class="k">[ldap]</span> <span class="na">accept</span> <span class="o">=</span> <span class="s">127.0.0.1:389</span> <span class="na">client</span> <span class="o">=</span> <span class="s">yes</span> <span class="na">connect</span> <span class="o">=</span> <span class="s">localhost:636</span> </pre></div> <p>We would run stunnel like this:</p> <div class="highlight"><pre><span class="n">stunnel</span> <span class="o">/</span><span class="n">path</span><span class="o">/</span><span class="n">to</span><span class="o">/</span><span class="n">stunnel</span><span class="p">.</span><span class="n">conf</span> </pre></div> <p>Again, I would run this under the control of a service supervisor. To keep stunnel in the foreground we would add the following to the global section of the configuration file (i.e., before the <code>[ldap]</code> section marker):</p> <div class="highlight"><pre><span class="n">foreground</span> <span class="o">=</span> <span class="n">yes</span> </pre></div> <p>With both of these solutions in place, we have achieved the following:</p> <ul> <li>High availability.</li> </ul> <p>Our application will transparently make use of multiple directory servers.
