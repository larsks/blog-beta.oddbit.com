This article is primarily about problems I encountered along the way, while the <a href="/post/openstack-windows-2">second article</a> focuses on the mechanism I eventually put in place.</p> <h2>Motivations</h2> <p>Like many organizations, we have a mix of Linux and Windows in our environment. Some folks in my group felt that it would be nice to let our Windows admins take advantage of OpenStack for prototyping and sandboxing in the same ways our Linux admins can use it.</p> <p>While it is trivial to get Linux instances running in OpenStack (there are downloadable images from several distributions that will magically configure themselves on first boot), getting Windows systems set up is a little trickier. There are no pre-configured images to download, and it looks as if there aren't that many people trying to run Windows under OpenStack right now so there is a lot less common experience to reference.</p> <h2>Like the cool kids do it</h2> <p>My first approach to this situation was to set up our Windows instances to act just like our Linux instances:</p> <ul> <li>Install <a href="http://cygwin.com/">Cygwin</a>.</li> <li>Run an SSH server.</li> <li>Have the system pull down an SSH public key on first boot and use this for administrative access.</li> </ul> <p>This worked reasonably well, but many people felt that this wasn't a great solution because it wouldn't feel natural to a typical Windows administrator. It also required a full Cygwin install to drive things, which isn't terrible but still feels like a pretty big hammer.</p> <p>As an alternative, we decided we needed some way to either (a) allow the user to pass a password into the instance environment, or (b) provide some way for the instance to communicate a generated password back to the user.</p> <h2>How about user-data?</h2> <p>One of my colleagues suggested that we could allow people to pass an administrative password into the environment via the <code>user-data</code> attribute available from the <a href="http://docs.openstack.org/trunk/openstack-compute/admin/content/metadata-service.html">metadata service</a>. While this sounds like a reasonable idea at first, it has one major flaw: data from the metadata service is available to anyone on the system who is able to retrieve a URL. This would make it trivial for anyone on the instance to retrieve the administrator password.</p> <h2>How about adminPass?</h2> <p>When you boot an instance using the nova command line tools...</p> <div class="highlight"><pre><span class="n">nova</span> <span class="n">boot</span> <span class="p">...</span> </pre></div> <p>You get back a chunk of metadata, including an <code>adminPass</code> key, which is a password randomly generated by OpenStack and availble during the instance provisioning process:</p> <div class="highlight"><pre><span class="o">+------------------------+--------------------------------------+</span> <span class="o">|</span> <span class="n">Property</span> <span class="o">|</span> <span class="n">Value</span> <span class="o">|</span> <span class="o">+------------------------+--------------------------------------+</span> <span class="p">...</span> <span class="o">|</span> <span class="n">adminPass</span> <span class="o">|</span> <span class="n">RBiWrSNYqK5R</span> <span class="o">|</span> <span class="p">...</span> <span class="o">+------------------------+--------------------------------------+</span> </pre></div> <p>This would be an ideal solution, if only I were able to figure out how OpenStack made this value available to the instance. After asking around on <a href="http://wiki.openstack.org/UsingIRC">#openstack</a> it turns out that not many people were even aware this feature exists, so information was hard to come by. I ran across some <a href="http://docs.openstack.org/trunk/openstack-compute/admin/content/hypervisor-configuration-basics.html">documentation</a> that mentioned the <code>libvirt_inject_password</code> option in <code>nova.conf</code> with the following description:</p> <blockquote> <p>(BoolOpt) Inject the admin password at boot time, without an agent.</p> </blockquote> <p>...but that still didn't actually explain how it worked, so I went diving through the code. The <code>libvirt_inject_password</code> option appears in only a single file, <code>nova/virt/libvirt/connection.py</code>, so I knew where to start. This led me to the <code>_create_image</code> method, which grabs the <code>admin_pass</code> generated by OpenStack:</p> <div class="highlight"><pre><span class="k">if</span> <span First, it's possible that the host will be unable to mount the NTFS filesystem, and secondly, there are no <code>passwd</code> or <code>shadow</code> files of any use on the target.</p> <p>You can pass <code>--config-drive=True</code> to <code>nova boot</code> and it will use a configuration drive (a whole-disk FAT filesystem) for configuration data (and make this available as a block device when the system boots), but this fails, hard: most of the code treats this as being identical to the original root filesystem, so it still tries to perform the modifications to <code>/etc/passwd</code> and <code>/etc/shadow</code> which, of course, don't exist.</p> <p>I whipped some quick <a href="https://github.com/seas-computing/nova/commits/lars/admin_pass">patches</a> that would write the configuration data (such as <code>admin_pass</code>) to simple files at the root of the configuration drive...but then I ran into a new problem:</p> <p>Windows doesn't know how to deal with whole-disk filesystems (nor, apparently, do many <a href="http://serverfault.com/questions/444446/mounting-whole-disk-filesystems-in-windows-2008/444448#comment481758_444448">windows admins</a>). In the absence of a partition map, Windows assumes that the device is empty.</p> <p>Oops. At this point it was obvious I was treading on ground best lest undisturbed.</p> <p>This ends my tail of woe; tune in next...um, article...in which I discuss the solution we're using.</p>Lars PowerShellhttp://blog-beta.oddbit.com/posts/2012/11/04/powershell-wait-for-networking/<p>I've recently been exploring the world of Windows scripting, and I ran into a small problem: I was running a script at system startup, and the script was running before the network interface (which was using DHCP) was configured.</p> <p>There are a number of common solutions proposed to this problem:</p> <ul> <li> <p>Just wait for some period of time.</p> <p>This can work but it's ugly, and because it doesn't actually verify the network state it can result in things breaking if some problem prevents Windows from pulling a valid DHCP lease. - Use ping to verify the availability of some remote host.</p> <p>This works reasonably well if you have a known endpoint you can test, but it's hard to make a generic solution using this method.</p> </li> </ul> <p>What I really wanted to do was to have my script wait until a default gateway appeared on the system (which would indicate that Windows had successfully acquired a DHCP lease and had configured the interface).</p> <p>My first attempts involved traditional batch scripts (<code>.bat</code>) running some variant of <code>route print</code> and searching the output. This can work, but it's ugly, and I was certain there must be a better way. I spent some time learning about accessing network configuration information using PowerShell, and I came up with <a href="https://gist.github.com/4011808">the following code</a>:</p> <div class="highlight"><pre><span class="err">#</span> <span class="nb">Wait</span> <span class="nb">for</span> <span class="nx">a</span> <span class="nx">DHCP</span><span class="na">-enabled</span> <span class="nx">interface</span> <span class="k">to</span> <span class="nx">develop</span> <span class="err">#</span> <span class="nx">a</span> <span class="nb">default</span> <span class="nx">gateway.</span> <span class="err">#</span> <span class="err">#</span> <span class="nb">usage</span><span class="p">:</span> <span class="nb">wait</span><span class="na">-for-network</span> <span class="err">[</span> <span class="o">&lt;</span><span class="nx">tries</span><span class="o">&gt;</span> <span class="cp">]</span> function wait-for-network ($tries) { while (1) { # Get a list of DHCP-enabled interfaces that have a # non-$null DefaultIPGateway property. $x = gwmi -class Win32_NetworkAdapterConfiguration ` -filter DHCPEnabled=TRUE | where { $_.DefaultIPGateway -ne $null } # If there is (at least) one available, exit the loop. if ( ($x | measure).count -gt 0 ) { break } # If $tries &gt; 0 and we have tried $tries times without # success, throw an exception. if ( $tries -gt 0 -and $try++ -ge $tries ) { throw &quot;Network unavaiable after $try tries.&quot; } # Wait one second. start-sleep -s 1 } } </pre></div> <p>This uses various sort of filtering to get a list of DHCP-enabled interfaces that have a default gateway (the <code>DefaultIPGateway</code> attribute). It will poll the state of things once/second up to <code>$tries</code> times, and if nothing is available it will ultimately throw an exception.</p>Lars Kellogg-StedmanSun, 04 Nov 2012 00:00:00 -0400tag:blog-beta.oddbit.com,2012-11-04:posts/2012/11/04/powershell-wait-for-networking/powershellnetworkingwindows
