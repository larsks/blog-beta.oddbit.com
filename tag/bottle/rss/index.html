This project can be deployed directly on <a href="http://openshift.com/">OpenShift</a> if you want to try things out on your own. You can also try it out online at <a href="http://pubsub.example.oddbit.com/">http://pubsub.example.oddbit.com/</a>.</p> <!-- more --> <h2>What is long polling?</h2> <p>Long polling is a technique used in web applications to enable a low-latency response to server messages without the CPU or traffic overhead of high frequency polling.</p> <p>A client makes a request to the web server, but rather than responding immediately the server holds the connection (for a potentially lengthy time), and only response when data is available. The client will react to this data and then restart the poll and wait for more data.</p> <h2>Long polling with Jquery</h2> <p>I've opted to use <a href="http://jquery.com/">jquery</a> as part of my client-side implementation, because I've used it a little in the past and it simplifies things a great deal.</p> <p>There are a number of articles out there that describe the client-side implementation of long polling using jquery. <a href="http://techoctave.com/c7/posts/60-simple-long-polling-example-with-javascript-and-jquery">This</a> article from Seventh Octave gives what is a typical example:</p> <div class="highlight"><pre><span class="p">(</span><span class="kd">function</span> <span class="nx">poll</span><span class="p">(){</span> <span class="nx">$</span><span class="p">.</span><span class="nx">ajax</span><span class="p">({</span> <span class="nx">url</span><span class="o">:</span> <span class="s2">&quot;/poll&quot;</span><span class="p">,</span> <span class="nx">success</span><span class="o">:</span> <span class="kd">function</span> <span class="p">(</span><span class="nx">data</span><span class="p">)</span> <span class="p">{</span> <span class="nx">do_something_with</span><span class="p">(</span><span class="nx">data</span><span class="p">);</span> <span class="p">},</span> <span class="nx">dataType</span><span class="o">:</span> <span class="s2">&quot;json&quot;</span><span class="p">,</span> <span class="nx">complete</span><span class="o">:</span> <span class="nx">poll</span><span class="p">,</span> <span class="nx">timeout</span><span class="o">:</span> <span class="mi">30000</span> <span class="p">});</span> <span class="p">})();</span> </pre></div> <p>This defines a function <code>poll()</code> that gets called automatically. Because I'm not a JavaScript person it took me a moment to figure out exactly how that worked. So that you are less mysified than I: The basic structure of this function is:</p> <div class="highlight"><pre><span class="p">(</span><span class="kd">function</span> <span class="nx">poll</span><span class="p">()</span> <span class="p">{...})();</span> </pre></div> <p>Since the <code>function</code> keyword returns a value (a reference to the defined function), we can call that reference immediately. This construct is entirely equivalent to:</p> <div class="highlight"><pre><span class="kd">function</span> <span class="nx">poll</span><span class="p">()</span> <span class="p">{...}</span> <span class="nx">poll</span><span class="p">();</span> </pre></div> <p>When called, it fires off an asynchronous AJAX request to <code>/poll</code> on our server. If the server sends a response, the browser will call the function in the <code>success</code> attribute, which will presumably do something useful with the response data.</p> <p>It's important not to gloss over the fact that the <a href="http://api.jquery.com/jQuery.ajax/">ajax</a> method is called asynchronously. The <code>poll()</code> function returns immediately when it is called, allowing your client to continue processing your script(s).</p> <p>The callable in the <code>complete</code> attribute will be called at the end of the AJAX request, regardless of whether or not the request was successful. The use of the <code>timeout</code> attribute ensures that this code will not poll more frequently than once every 30 seconds (this helps prevent the code from monopolizing the CPU by polling too frequently if an error is causing the AJAX request to return immediately).</p> <h2>A simple web page</h2> <p>Our chat application is going be built around a very simple web page with two fields (one for a "nickname", and one for entering messages to send) and a container for printing messages received from the server. <code>POST</code> request and publish a JSON message onto the message bus.</p> <h2>Writing the server: sending messages</h2> <p>Having received a message from a client, our task is to send that Because we're using 0MQ's publish-and-subscribe support, any message sent on <code>pubsocket</code> will automatically be propagated to any connected subscribers.</p> <p>When each client requests <code>/sub</code>, Bottle runs this function, which blocks waiting for messages on the message bus. When a message arrives, the function returns it to the client and exits. Note there's a little bit of magic here: when a request handler returns a dictionary to Bottle, Bottle automatically serializes that as JSON before sending the response to the client.</p> <h2>Fixing things up</h2> <p>While the above code seems to work, there is a potential problem with the implementation: If a client that has been waiting on <code>/sub</code> disconnects, the <code>sub</code> function will continue to remain blocked on the call to <code>subsock.recv_json</code>. This means that the server will hold open the file descriptor associated with the connection. Once a message is received, the server will attempt to send a response, notice that the client has disconnected, and close the file descriptor. Given a large enough population of clients, the number of open file descriptors could run into system resource limits. In order to prevent this situation, we need to react to client disconnects...which means that now, instead of just blocking waiting for messages, we <em>also</em> need to wait for some notification that a client has disconnected.</p> <p>In a traditional socket program, you might do this with something like the <code>poll()</code> or <code>select()</code> function. Since we're using 0MQ...we'll do exactly the same thing, which is one of the reasons 0MQ is fun to work with. We first need to figure out how to detect client disconnects. The <a href="http://www.python.org/dev/peps/pep-0333/">WSGI specification</a> doesn't provide a standard way to expose the client socket to our application. However, inspection of the WSGI environment (in <code>bottle.request.environ</code> reveals that the <code>wsgi.input</code> member contains an <code>rfile</code> attribute, which is exactly what we need.
