I'm using GitHub Pages to publish everything, so it seemed easiest to make tag generation part of a <code>pre-push</code> hook (new in Git 1.8.2). This hook is run automatically as part of the <code>git push</code> operation, so it's the perfect place to insert generated content that must be kept in sync with posts on the blog.</p> <!-- more --> <h2>Keeping things in sync</h2> <p>The <code>_posts</code> directory of my blog is a <a href="http://git-scm.com/book/en/Git-Tools-Submodules">git submodule</a>, which means it gets updated and pushed asynchronously with respect to the main repository. We want to make sure that we don't regenerate the tag feeds if there are either uncomitted changes in <code>_posts</code> <em>or</em> if there are <em>unpushed</em> changes in <code>_posts</code>: in either situation, we could generate a tag feed for tags that weren't actually used in any published posts.</p> <p>The following checks for any uncomitted changes in <code>_posts</code>:</p> <div class="highlight"><pre><span class="k">if</span> <span class="o">!</span> <span class="n">git</span> <span class="n">diff</span><span class="o">-</span><span class="n">files</span> <span class="o">--</span><span class="n">quiet</span> <span class="n">_posts</span><span class="p">;</span> <span class="n">then</span> <span class="n">echo</span> <span class="s">&quot;posts are out of sync (skipping tag maintenance)&quot;</span> <span class="n">exit</span> <span class="mi">0</span> <span class="n">fi</span> </pre></div> <p>This will abort the tag feed generation if any of the following is true:</p> <ul> <li><code>_posts</code> has uncomitted changes</li> <li><code>_posts</code> has new, untracked content</li> <li><code>_posts</code> is at a revision that differs from the last comitted revision in the parent repository.</li> </ul> <p>This still leaves one possible failure mode: if we commit all changes in <code>_posts</code>, and then commit the updated <code>_posts</code> revision in the parent repository, all of the previous checks will pass...but since we haven't pushed the <code>_posts</code> repository, we could still be pushing tags that don't match up with published posts.</p> <p>The following check will prevent this situation by checking if the repository differs from the upstream branch:</p> <div class="highlight"><pre><span class="k">if</span> <span class="o">!</span> <span class="p">(</span><span class="n">cd</span> <span class="n">_posts</span><span class="p">;</span> <span class="n">git</span> <span class="n">diff</span><span class="o">-</span><span class="n">index</span> <span class="o">--</span><span class="n">quiet</span> <span class="n">origin</span><span class="o">/</span><span class="n">posts</span><span class="p">);</span> <span class="n">then</span> <span class="n">echo</span> <span class="s">&quot;posts are out of sync (skipping tag maintenance)&quot;</span> <span class="n">exit</span> <span class="mi">0</span> <span class="n">fi</span> </pre></div> <h2>Generating tag feeds</h2> <p>In order to prevent stale tags, we need to delete and regenerate all the tag feeds. Cleaning up the existing tag feeds is taken care of by the <code>cleantagfeeds</code> script:</p> <div class="highlight"><pre><span class="n">echo</span> <span class="s">&quot;cleaning tag feeds&quot;</span> <span class="n">_oddbit</span><span class="o">/</span><span class="n">cleantagfeeds</span> </pre></div> <p>Which is really just a wrapper for the following <code>find</code> commands:</p> <table class="highlighttable"><tr><td class="linenos"><div class="linenodiv"><pre>1 2 3 4 5 6 7</pre></div></td><td class="code"><div class="highlight"><pre><span class="c">#!/bin/sh</span> <span class="c"># Delete tag feeds unless there is a `.keep` file in the</span> <span class="c"># same directory.</span> find tag/* -name index.xml <span class="se">\</span> -execdir sh -c <span class="s1">&#39;test -f .keep || rm -f index.xml&#39;</span> <span class="se">\;</span> find tag/* -type d -delete </pre></div> </td></tr></table> <p>This will preserve any tag feeds that have a corresponding <code>.keep</code> file (just in case we've done something special that requires manual intervention) and deletes everything else.</p> <p>Generating the tag feeds is taken care of by the <code>gentagfeeds</code> script:</p> <div class="highlight"><pre><span class="n">echo</span> <span class="s">&quot;generating tag feeds&quot;</span> <span class="n">_oddbit</span><span class="o">/</span><span class="n">gentagfeeds</span> </pre></div> <p>This is a Python program that iterates over all the files in <code>_posts</code>, reads in the YAML frontmatter from each one, and then generates a feed file for each tag using a template.</p> <p>Finally, we need to add any changes to the repository. We unilaterally add the <code>tags/</code> directory:</p> <div class="highlight"><pre><span class="n">git</span> <span class="n">add</span> <span class="o">-</span><span class="n">A</span> <span class="n">tag</span> </pre></div> <p>And then see if that got us anything:</p> <div class="highlight"><pre><span class="k">if</span> <span class="o">!</span> <span class="n">git</span> <span class="n">diff</span><span class="o">-</span><span class="n">index</span> <span class="o">--</span><span class="n">quiet</span> <span class="n">HEAD</span> <span class="o">--</span> <span class="n">tag</span><span class="p">;</span> <span class="n">then</span> <span class="n">git</span> <span class="n">commit</span> <span class="o">-</span><span class="n">m</span> <span class="err">&#39;</span><span class="n">automatic</span> <span class="n">tag</span> <span class="n">update</span><span class="err">&#39;</span> <span class="n">tag</span> <span class="n">fi</span> </pre></div> <p>At this point, we've regenerated all the tag feeds and committed any new or modified tag feeds to the repository, which will get published to GitHub as part of the current <code>push</code> operation.</p> <p>The actual feed templates look like this:</p> <div class="highlight"><pre><span class="o">---</span> <span class="nl">layout:</span> <span class="n">rss</span> <span class="nl">exclude:</span> <span class="nb">true</span> <span class="nl">tags:</span> <span class="o">-</span> <span class="p">{{</span><span class="n">tag</span><span class="p">}}</span> <span class="o">---</span> </pre></div> <p>I'm using a modified version of <a href="https://github.com/thedereck/gh-pages-blog/">gh-pages-blog</a> in which I have modified <code>_layouts/rss.xml</code> to optionally filter posts by tag using the following template code:</p> <p _="%" endfor="endfor">{% raw %} . . . {% for p in site.posts %} {% if page contains 'tags' %} {% assign selected = false %} {% <span class="n">selected</span> <span class="o">==</span> <span class="nb">false</span> <span class="o">%</span><span class="p">}</span> <span class="p">{</span><span class="o">%</span> <span class="p">}</span> <span class="p">{</span><span class="o">%</span> <span class="n">endif</span> <span class="o">%</span><span class="p">}</span> <span class="p">.</span> <span class="p">.</span> <span class="p">.</span> </pre></div> <p>{% endraw %}</p> <p>For each post on the site (<code>site.posts</code>), this checks for any overlap between the tags in the post and the tags selected in the tag feed. While the automatic feeds use only a single tag, this also makes it possible to create feeds that follow multiple tags.</p> <p>All of the code used to implement this is available in the <a href="http://github.com/larsks/blog.oddbit.com/">GitHub repository for this blog</a>.</p>Lars Kellogg-StedmanFri, 22 Nov 2013 00:00:00 -0500tag:blog-beta.oddbit.com,2013-11-22:posts/2013/11/22/automatic-maintenanc/bloggitEnabled blog commentshttp://blog-beta.oddbit.com/posts/2013/11/18/enabled-comments/<p>I've enabled blog comments using <a href="http://disqus.com/">Disqus</a>. Unfortunately, while I like the publish-via-Dropbox mechanism, there have been enough problems recently that I've finally switched over to using <a href="http://pages.github.com/">GitHub Pages</a> for hosting. I've been thinking about doing this for a while, but the things that finally pushed me to make the change were:</p> <ul> <li>Sync problems that would prevent new posts from appearing (and that at least once caused posts to disappear).</li> <li>Lack of any response to bug reports by the site maintainers.</li> </ul> <p>A benefit of the publish-via-Dropbox mechanism is, of course, that I already had all the data and didn't need to go through any sort of export process.</p> <!-- more --> <h2>Fixing metadata</h2> <p>Like <a href="http://scriptogr.am/">scriptogram</a>, <a href="http://pages.github.com/">GitHub Pages</a> is also a Markdown-based solution.
